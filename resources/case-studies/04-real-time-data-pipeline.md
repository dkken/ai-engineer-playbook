# 案例：Netflix 的实时推荐数据管道

> **来源**：Netflix 技术博客 (netflixtechblog.com)、ACM Queue 论文、RecSys 会议论文、QCon 技术分享
> **公司**：Netflix（全球最大流媒体平台，2025Q4 付费订阅用户 2.83亿，覆盖 190+ 国家）
> **场景**：实时个性化推荐
> **技术栈**：Apache Flink、Kafka、Spark、自研 ML 平台（Metaflow）

---

## 背景

Netflix 的推荐系统每天处理数十亿次用户交互，为 2.83 亿用户提供个性化内容推荐。推荐系统的核心是一个实时数据管道，处理用户行为、内容元数据、A/B 测试数据等。

**业务指标**（来源：Netflix 技术博客、公开财报）：
- 付费订阅用户：2.83 亿（2025Q4）
- 日活跃用户：约 1.5 亿+
- 日处理事件：数千亿
- 推荐系统贡献：约 80% 观看时长来自推荐（Netflix 公开披露）
- 内容库：数万部电影和剧集

---

## 需求

1. **实时性**：用户行为发生后，推荐模型需在秒级更新
2. **可扩展性**：支持全球 2.83 亿用户
3. **准确性**：推荐准确率直接影响用户留存
4. **多模态**：处理视频、音频、文本、图片等多种内容
5. **A/B 测试**：支持大规模在线实验

---

## 架构

### 核心设计决策

**1. Lambda 架构（批流结合）**

- **批处理层**：每日全量训练推荐模型，处理历史数据
- **速度层**：实时处理用户行为，更新在线特征
- **服务层**：合并批处理和实时结果，提供推荐服务

**2. 特征工程管道**

```
用户行为 → Kafka → Flink 实时特征 → 特征存储 → 推荐服务
内容元数据 → S3 → Spark 批处理特征 → 特征存储 → 推荐服务
```

**3. 模型服务**

- **在线模型**：轻量级模型，用于实时推荐
- **离线模型**：复杂模型，用于候选集生成
- **级联架构**：多阶段过滤（召回 → 粗排 → 精排）

### 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 消息队列 | Apache Kafka | 用户行为事件流 |
| 流处理 | Apache Flink | 实时特征计算 |
| 批处理 | Apache Spark | 离线模型训练 |
| 特征存储 | 自研 + Redis | 在线特征查询 |
| 模型服务 | 自研 ML 平台 | 模型部署和推理 |
| 数据仓库 | Apache Iceberg | 数据湖 |
| ML 工作流 | Metaflow（开源） | Netflix 自研并开源的 ML 工作流框架 |

---

## 挑战与解决方案

### 挑战 1：冷启动

**问题**：新用户或新内容没有历史数据，无法生成推荐。

**解决方案**：
- **内容冷启动**：基于内容元数据（类型、演员、导演）的相似度推荐
- **用户冷启动**：基于人口统计信息和流行度推荐
- **探索机制**：Epsilon-greedy 策略，一定比例随机推荐

### 挑战 2：数据倾斜

**问题**：热门内容和活跃用户的数据量远大于长尾，导致处理倾斜。

**解决方案**：
- **分区策略**：按用户 ID 分区，确保负载均衡
- **采样策略**：对热门内容进行下采样
- **局部聚合**：在分区内部先聚合，减少数据传输

### 挑战 3：模型更新与一致性

**问题**：离线训练的模型与在线特征可能不一致。

**解决方案**：
- **特征版本管理**：每个模型绑定特定版本的特征
- **影子测试**：新模型上线前先影子测试
- **渐进式发布**：灰度发布，逐步切换流量

## 面试要点

### 面试官考察维度

1. **实时推荐架构**
   - 如何设计一个支持 2.83 亿用户的推荐系统？
   - Lambda 架构 vs Kappa 架构的 trade-off？
   - 如何平衡实时性和模型复杂度？

2. **特征工程**
   - 实时特征和离线特征如何保持一致？
   - 特征存储如何设计？
   - 如何处理特征漂移？

3. **A/B 测试**
   - 推荐系统的 A/B 测试如何设计？
   - 如何评估推荐效果？
   - 多目标优化（点击率、观看时长、留存）如何处理？

### 参考追问

- "如果用户刚看完一部电影，推荐系统应该在多久后更新推荐？"
- "如何处理推荐系统的'信息茧房'问题？"
- "推荐模型的特征数量达到 10 万，如何优化特征存储和查询？"
