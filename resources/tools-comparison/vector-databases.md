# 向量数据库选型对比

> **使用场景**：RAG 系统、语义搜索、推荐系统、知识图谱
>
> **选型原则**：没有最好的，只有最适合的。根据数据规模、查询模式、部署环境和团队技术栈选择。

## 详细分析

### Milvus

**优势**：
- 专为向量检索设计，性能优异
- 支持十亿级向量
- 丰富的距离度量（L2、IP、余弦、汉明等）
- 强大的分布式架构
- 活跃的社区和生态

**劣势**：
- 部署相对复杂
- 学习曲线较陡
- 需要额外维护（etcd、MinIO、Pulsar）

**适用场景**：
- 大规模向量数据（千万级以上）
- 需要分布式部署
- 对性能要求极高

**实战项目**：P1-01、P2-02、P3-01

---

### Weaviate

**优势**：
- 原生支持混合检索（向量 + BM25）
- 内置向量化模块（可选）
- GraphQL 接口友好
- 模块化设计，可扩展

**劣势**：
- 分布式版本为商业版
- 大规模数据性能不如 Milvus

**适用场景**：
- 需要混合检索
- 中等规模数据（百万级）
- 喜欢 GraphQL

**实战项目**：P2-02

---

### Pinecone

**优势**：
- 完全托管，零运维
- 性能极高，延迟低
- 自动扩缩容
- 混合检索支持

**劣势**：
- 商业产品，成本高
- 无法自托管
- 数据锁定风险

**适用场景**：
- 快速原型验证
- 无运维团队
- 预算充足

**实战项目**：不推荐用于学习（无法本地部署）

---

### Chroma

**优势**：
- 极易上手，5分钟启动
- 纯 Python，无需额外依赖
- 适合原型开发
- 活跃的社区

**劣势**：
- 性能一般
- 不支持分布式
- 不适合生产环境大规模数据

**适用场景**：
- 快速原型验证
- 学习和教学
- 小规模数据（万级以下）

**实战项目**：P1-01（入门阶段）

---

### Qdrant

**优势**：
- Rust 实现，性能优异
- 支持过滤和混合检索
- 易于部署（单二进制文件）
- 云原生设计

**劣势**：
- 社区相对较小
- 生态不如 Milvus 丰富

**适用场景**：
- 中等规模数据
- 需要高性能
- 喜欢 Rust 技术栈

**实战项目**：P1-01、P2-02

---

### pgvector

**优势**：
- 基于 PostgreSQL，无需额外数据库
- 事务支持
- 丰富的 SQL 生态
- 与现有系统无缝集成

**劣势**：
- 向量性能不如专用数据库
- 维度限制（16,000）
- 不适合纯向量场景

**适用场景**：
- 已有 PostgreSQL 基础设施
- 需要事务和关系数据
- 中小规模向量数据

**实战项目**：P3-02

## 选型决策树

```
数据规模？
├── < 10万 → Chroma（快速原型）或 pgvector（已有PG）
├── 10万-1000万 → Qdrant 或 Weaviate
└── > 1000万 → Milvus

是否需要混合检索？
├── 是 → Weaviate、Milvus、Qdrant
└── 否 → 任意

是否需要自托管？
├── 是 → Milvus、Weaviate、Qdrant、Chroma、pgvector
└── 否 → Pinecone（预算充足）或 Zilliz（Milvus云）

团队技术栈？
├── Python为主 → Chroma、Qdrant
├── Go/Rust → Qdrant、Milvus
├── 已有PostgreSQL → pgvector
├── 喜欢GraphQL → Weaviate
```

## 面试考察点

### 常见问题

1. **为什么选择 Milvus 而不是 Chroma？**
   - 考察点：对性能、规模、运维的理解
   - 期望回答：数据规模、分布式需求、性能要求

2. **向量数据库和传统数据库的区别？**
   - 考察点：对向量检索原理的理解
   - 期望回答：ANN 算法、索引结构、距离度量

3. **如何评估向量数据库的性能？**
   - 考察点：对评估指标的了解
   - 期望回答：召回率、延迟、吞吐量、内存占用
