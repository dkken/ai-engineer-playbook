# 评估框架选型对比

> **使用场景**：LLM 输出质量评估、RAG 系统评估、Agent 性能评估、安全评估
>
> **选型原则**：根据评估目标（准确性、安全性、成本）、技术栈、部署需求选择。

## 详细分析

### Ragas

**优势**：
- 专为 RAG 设计，指标精准
- 与 LangChain 生态集成
- 社区活跃，更新频繁
- 学术论文支撑

**劣势**：
- 仅支持 RAG 场景
- 可视化能力弱
- 不支持安全评估

**适用场景**：
- RAG 系统评估
- 学术研究
- 快速验证 RAG 效果

**实战项目**：P1-01、P2-02

---

### DeepEval

**优势**：
- 50+ 评估指标
- 支持多种评估类型
- 易于集成到 CI/CD
- 文档完善

**劣势**：
- 可视化一般
- 社区相对较小

**适用场景**：
- 通用 LLM 评估
- 需要多种指标
- CI/CD 集成

**实战项目**：P1-03

---

### Opik

**优势**：
- 可观测性 + 评估一体化
- 50+ 评估指标
- Agent 优化器
- 成本追踪
- 生产级监控

**劣势**：
- 功能多，学习曲线较陡
- 自托管需要资源

**适用场景**：
- 生产环境评估
- 需要可观测性
- Agent 优化

**实战项目**：P1-03、P3-03

---

### Giskard

**优势**：
- 安全评估专业
- Red Teaming 支持
- 偏见检测
- 合规评估

**劣势**：
- 通用评估能力一般
- 学习曲线较陡

**适用场景**：
- AI 安全评估
- 合规检查
- Red Teaming

**实战项目**：P2-04

---

### Promptflow

**优势**：
- 微软出品，与 Azure 集成
- Prompt 版本管理
- 可视化编辑
- 与 VS Code 集成

**劣势**：
- 主要面向 Azure
- 开源版功能受限
- 社区相对较小

**适用场景**：
- Azure 生态
- Prompt 工程
- 快速原型

## 选型决策树

```
评估目标？
├── RAG 质量 → Ragas
├── 通用 LLM → DeepEval
├── 可观测性+评估 → Opik
├── 安全/合规 → Giskard
└── Prompt 工程 → Promptflow

是否需要生产监控？
├── 是 → Opik
└── 否 → 其他

是否需要 CI/CD 集成？
├── 是 → DeepEval、Opik、Ragas
└── 否 → 任意
```

## 面试考察点

### 常见问题

1. **如何评估 RAG 系统的质量？**
   - 考察点：对 RAG 评估指标的理解
   - 期望回答：Faithfulness、Answer Relevancy、Context Precision、Context Recall

2. **LLM-as-a-Judge 的局限性？**
   - 考察点：对评估方法的理解
   - 期望回答：偏见、成本、一致性、无法替代人工

3. **如何设计自定义评估指标？**
   - 考察点：对业务场景的理解
   - 期望回答：业务目标、数据特点、可解释性
