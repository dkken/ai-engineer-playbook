### 为什么选择 graphiti？

Graphiti 是 2026 年知识图谱领域的标杆开源项目，由 Zep 团队开发，核心差异化能力：

1. **时间感知（Temporal）**：双时间模型（事件发生时间 + 事实有效期），支持历史查询和事实失效检测
2. **增量更新**：无需全图重算，实时写入新数据
3. **混合检索**：BM25 关键词 + 向量语义 + 图遍历并行检索
4. **多后端支持**：Neo4j、FalkorDB、Kuzu、Amazon Neptune
5. **MCP 服务**：提供 Model Context Protocol 标准接口，可被 Claude、Cursor 等直接调用
6. **生产验证**：Zep 平台核心引擎，已在金融、医疗、法律领域大规模部署

**与其他知识图谱方案对比**：

| 特性 | Graphiti | Mem0 | Letta | 传统 GraphRAG |
|------|----------|------|-------|--------------|
| 时间感知 | ✅ 双时间模型 | ❌ 无 | ❌ 无 | ❌ 无 |
| 增量更新 | ✅ 实时 | ✅ 实时 | ✅ 实时 | ❌ 批处理 |
| 混合检索 | ✅ 三种并行 | ❌ 向量 only | ❌ 分页 | ❌ 图遍历 |
| 事实失效检测 | ✅ 自动 | ❌ 共存 | ❌ 共存 | ❌ 手动 |
| 实体解析 | ✅ LLM+确定性 | ✅ LLM | ✅ LLM | ❌ 简单匹配 |
| 开源协议 | Apache-2.0 | Apache-2.0 | MIT | 多种 |

## 工程化改造任务

### 任务 1：部署与配置（2 小时）
- 使用 Docker 部署 Neo4j 图数据库
- 配置 Graphiti 连接 Neo4j
- 配置本地 LLM（Ollama）替代默认 API
- 配置 Embedding 模型（本地或 API）

### 任务 2：知识图谱构建（4 小时）
- 使用 browser-use 采集目标网站数据
- 使用 Graphiti 的 Episode 机制写入数据
- 实现实体和关系提取
- 配置自定义实体类型（Pydantic 定义）
- 验证图谱构建结果

### 任务 3：RAG 集成（4 小时）
- 集成 kotaemon 作为 RAG 前端
- 实现 Graphiti 混合检索作为 kotaemon 的检索后端
- 配置检索策略（语义 + BM25 + 图遍历权重）
- 实现多轮对话上下文记忆
- 添加回答引用来源标注（图遍历路径）

### 任务 4：实时更新与优化（2 小时）
- 实现数据变更监听（文件系统/Webhook）
- 配置 Graphiti 增量更新
- 设计评估指标（检索准确率、回答相关性、图谱覆盖率）
- 对比向量 RAG vs 知识图谱 RAG 的效果

### 任务 5：生产化改造（2 小时）
- 添加错误处理和日志记录
- 配置环境变量管理（不硬编码 API Key）
- 实现图谱备份与恢复
- 编写部署文档和 README

## 常见坑点

- ❌ **坑点 1**：忽略时间感知配置，导致事实冲突无法处理
  - ✅ **解决**：正确配置 `t_valid` 和 `t_invalid` 字段
- ❌ **坑点 2**：实体解析不准确，导致同一实体被重复创建
  - ✅ **解决**：利用 Graphiti 的 MinHash + LSH 去重机制
- ❌ **坑点 3**：图遍历过深导致检索延迟过高
  - ✅ **解决**：限制遍历深度，配置超时机制
- ❌ **坑点 4**：不评估图谱质量，盲目扩展
  - ✅ **解决**：建立图谱覆盖率、实体一致性等评估指标
