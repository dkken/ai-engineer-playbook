## 工程化改造任务

### 任务 1：部署与配置（30 分钟）
- 使用 Docker 部署 opik
- 配置本地 LLM（Ollama）替代默认 API
- 配置评估指标（Faithfulness、AnswerRelevancy、ContextualRelevancy）

### 任务 2：功能扩展（1 小时）
- 集成 giskard-oss 添加安全测试能力
- 实现自定义评估指标
- 添加评估报告生成
- 实现批量评估功能

### 任务 3：评估与优化（1 小时）
- 设计评估数据集
- 对比不同评估指标的效果
- 优化评估流程

### 任务 4：生产化改造（30 分钟）
- 添加错误处理和日志记录
- 配置环境变量管理（不硬编码 API Key）
- 编写部署文档和 README

## 面试官考察维度

### 维度 1：评估指标设计能力（权重 40%）
- **考察点**：如何设计有意义的评估指标？
- **追问示例**：
  - "Ragas的Faithfulness和AnswerRelevancy有什么区别？什么场景下Faithfulness高但Relevancy低？"
  - "如果业务目标是降低客服人力成本，应该关注什么指标？（解决率、用户满意度、平均处理时间）"
  - "LLM-as-a-Judge的局限性是什么？什么情况下不能依赖自动评估？"

### 维度 2：安全评估能力（权重 35%）
- **考察点**：如何进行 Red Teaming？
- **追问示例**：
  - "giskard-oss的偏见检测和garak的漏洞扫描有什么区别？什么时候用哪个？"
  - "你设计了一个安全护栏，怎么验证它真的有效？（用对抗样本测试）"
  - "安全评估通过了，但上线后还是出现了有害输出，可能是什么原因？"

### 维度 3：成本与效率平衡（权重 25%）
- **考察点**：评估本身的成本如何控制？
- **追问示例**：
  - "用GPT-4做评估 judge，每次评估成本$0.05，1000条测试数据就是$50，怎么降低成本？"
  - "评估延迟很高（每次10秒），怎么优化到可接受的范围（<1秒）？"
  - "什么情况下可以用本地小模型替代GPT-4做评估？"

## 常见坑点
- ❌ **坑点 1**：直接用默认配置部署，不配置本地模型，导致 API 费用过高
  - ✅ **解决**：配置 Ollama 本地 LLM（如 Qwen2.5）做评估 judge
- ❌ **坑点 2**：评估指标设计不合理，导致结果不准确
  - ✅ **解决**：根据业务场景选择指标（客服场景关注解决率，搜索场景关注相关性）
- ❌ **坑点 3**：忽略安全评估，只关注准确性
  - ✅ **解决**：集成 giskard-oss 进行安全测试，覆盖偏见、幻觉、注入攻击
- ❌ **坑点 4**：评估数据与真实数据分布不一致，导致"实验室效果好，上线效果差"
  - ✅ **解决**：定期用真实用户反馈数据更新评估集，监控线上指标与离线指标的差距
